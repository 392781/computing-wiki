In this tutorial, we will set up Docker for your local machine.

Prereqs

Windows users:

  • Hyper-V enabled through BIOS
  • Windows Subsystem for Linux 2 (WSL2)

Download

First, we need to download the latest version of Docker. This can be done by visiting the link and downloading an installer for your system.

Install

MacOS

Follow the installation instructions on the .dmg file you downloaded

Linux

Follow the instructions found on this installation page.

Windows

Windows requires additional steps prior to the Docker installation.

  1. Install WSL2 (follow this link to the Microsoft Store):

  1. Once the download is complete, click "Open". This should launch a terminal session. You've successfully installed WSL2!

  1. We can turn our attention to installing Docker Desktop. Go into your downloads and open up the Docker Desktop Installer. This will launch a window. Make sure that "Use WSL 2 instead of Hyper-V" is selected!:

Congrats! Docker is now installed on your machine! To verify, you can open terminal and run docker -v to verify that it's installed:

In case WSL 2 doesn't work, another option is to enable Hyper-V in your BIOS instead.

Note: The following instructions will vary a bit depending on your computer's manufacturer. The overall process should be similar though.

  1. Reboot your PC and pay attention to the boot up screen. You will need to press a key (usually Enter, Esc, or Delete) to interrupt your startup. From there, select and enter your BIOS:

  1. Navigate your BIOS and look for something akin to "Security" or "Features". Under that tab you should find an option for "Virtualization" or "Hyper-V":

  1. Inside this menu you'll find options to enable your Hyper-V/Virtualization for your CPU. Enable them:

  1. Make sure to save and reset your computer:

  1. Make sure that when installing Docker Desktop, you disable the "Use WSL 2 instead of Hyper-V" option!

Now that Docker is installed locally, you can follow the Dev Containers with VS Code to install the necessary extensions (if you haven't already) and run development containers outside of a remote environment.
